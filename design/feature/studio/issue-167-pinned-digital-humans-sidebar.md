## 1. 背景与目标

### 1.1 背景

用户在首页 / 会话中与多个**数字员工**（Decision Agent，前端与路由中常以 `employee` 查询参数、与 dip-studio **数字员工 id** 对齐）协作时，重复从选择器或列表中切换成本较高。Issue 要求支持将常用员工**固定到侧边栏**，实现**快速切换**。

### 1.2 目标（可验收）

1. 用户可将指定数字员工加入侧栏「固定」列表（_entry point 见 §6_）。
2. 侧栏展示固定项（至少 **图标 + 名称**，名称过长截断）；布局与现有 Studio 侧栏区块（如工作计划、历史会话）**视觉层级一致**。
3. 点击某一固定项：跳转到会话能力，并带上对应员工标识，语义为**使用该员工的新会话上下文**（详见 §5.2）。
4. 用户可**取消固定**；列表与持久化一致。
5. 偏好**随账号持久化**、可多终端一致（依赖 §4 后端方案）。

### 1.3 非目标（本期不做）

- 侧栏内**拖拽排序**（若未来要做，需单独开需求；本期顺序以服务端返回列表顺序为准）。
- 在侧栏直接展示**未读数 / 运行状态**等富状态（除非与现有 DipChatKit 能力无冲突且产品有明确要求）。
- 修改 dip-studio **数字员工业务能力**本身（仅消费其 id 与列表/详情元数据）。

## 2. 术语与约定

| 术语 | 含义 |
|------|------|
| 数字员工 ID | 与 `/api/dip-studio/v1/digital-human` 等资源中的员工主键一致；会话路由里常映射为查询参数 `employee`。 |
| 固定 / 钉选 | 写入用户偏好中的 `pinned_digital_human_ids` 列表，并在侧栏展示。 |
| 新开会话 | 跳转时不携带既有 `sessionKey`，由 DipChatKit 按所选员工起新会话（与首页提交进入对话的行为一致）。 |

## 3. 现状与可复用模式（约束）

| 域 | 现状要点 | 对本期设计的约束 |
|----|----------|------------------|
| 会话入口 | `Home` → ` /studio/conversation?employee=…`；`Conversation` 将 `employee` 传给 `DipChatKit`。 | 固定项点击应对齐同一参数契约。 |
| 侧栏 | `HomeSider` / `AdminSider` 在具备 studio 模块时结构对称；含 `StudioMenuSection`、工作计划块、历史会话块等。 | 新分区需**两处同时接入**，避免管理员与普通入口 UI 分叉。 |
| 微应用钉选 | `StoreMenuSection` + `preferenceStore`：列表来自 applications 接口过滤 `pinned`，单项变更走 `PUT …/applications/{key}/pinned`。 | **数字员工钉选不能复用该接口**（资源维度不同）；但**交互范式**（钉/拔钉、侧栏展示、token 后拉取一次）可对齐。 |

## 4. 持久化方案（已定稿）

### 4.1 决策

采用 **dip-hub 用户偏好 API + 库表**，以保证**按用户隔离**、可扩展其它偏好字段。

> 说明：应用表上的 `pinned` 字段表示**安装实例级**的微应用钉选状态，与「每个登录用户_own 的数字员工列表」域模型不同；数字员工固定必须走**用户维度**存储。

### 4.2 逻辑存储模型

- **主键**：`user_id`（与登录后用户信息主键一致，通常 UUID 字符串）。
- **内容**：JSON 文档，当前版本仅规范键 **`pinned_digital_human_ids`**：`string[]`。
- **顺序**：数组顺序即侧栏**自上而下**展示顺序；**前端不得擅自重排后写回**除非产品提供排序交互。

### 4.3 接口契约（逻辑）

以下描述**契约语义**，具体错误码以 OpenAPI / 实现为准：

| 方法 | 路径 | 语义 |
|------|------|------|
| `GET` | `/api/dip-hub/v1/user/preferences` | 返回当前用户的偏好；若无记录，等价于 `pinned_digital_human_ids: []`。 |
| `PUT` | `/api/dip-hub/v1/user/preferences` | **全量替换** `pinned_digital_human_ids`（请求体与响应体形相同）。 |

**请求/响应体（逻辑）**：

```json
{
  "pinned_digital_human_ids": ["<id-1>", "<id-2>"]
}
```

**服务端校验（逻辑）**：

- 列表**有序去重**；元素 trim；单元素长度上限、列表条数上限由服务端强制（防止恶意 payload）。
- 服务端**不**校验 id 是否仍存在于 dip-studio（避免 hub 强依赖 studio 实时列表）；**展示层**负责过滤无效 id（§7）。

**客户端更新模式**：

- 「增加固定」：先 `GET` → 将 id 插入期望位置（如尾部）→ `PUT`  Whole list。
- 「取消固定」：先 `GET` → 过滤掉 id → `PUT`。
- 需处理**并发覆盖**：若未来冲突频繁，可考虑版本号或 PATCH；**本期**以「最后写入为准」为已知限制，可在文中保留为风险。

### 4.4 拉取时机（逻辑）

- 用户已登录且具备 studio 能力时：**每个登录会话（token）内至少拉取一次**偏好列表（可与现有「钉选微应用列表」拉取策略对齐：避免路由抖动重复请求）。
- **窗口聚焦**是否刷新：可选；若刷新，需防抖并与工作计划/历史会话刷新策略协调，避免侧栏 N+1。

## 5. 核心业务流程

### 5.1 进入系统后（侧栏有固定项）

1. 前端拉取 `GET /user/preferences` 得 id 列表。
2. 前端拉取或复用 `getDigitalHumanList()`（及必要时详情）解析 **name / icon**，与 id join；**仅在展示层**丢弃列表中不存在的 id（可选：异步写回「清理后的列表」以减少脏数据，属优化项需评估写放大）。
3. 渲染 `PinnedDigitalHumansSection`（命名可调整）。

### 5.2 用户点击侧栏某一固定项

1. `navigate('/studio/conversation?employee=<id>')`，**不带** `sessionKey`（新开会话语义）。
2. `Conversation` / `DipChatKit` 既有逻辑接管；若当前已在会话页，仅 `employee` 变化，应仍能切换员工上下文（以现有实现为准， regress）。

### 5.3 用户点击「固定当前员工」

1. 取得当前员工 id（来自会话状态或选择器）。
2. `GET` 偏好 → 若已存在则幂等成功 → 否则追加 → `PUT`。
3. 成功提示；失败按统一 HTTP 错误展示。

### 5.4 用户「取消固定」

1. `GET` → 移除 id → `PUT`。
2. 侧栏项即时消失（乐观更新可选：需失败回滚策略）。

## 6. 前端模块划分（逻辑）

| 模块 | 职责 |
|------|------|
| 状态层 | 缓存 `pinned_digital_human_ids`；封装 fetch / pin / unpin；与 token 生命周期对齐。 |
| 侧栏区块组件 | 负责展示列表、空态隐藏、折叠态与 `WorkPlanSection` 等一致。 |
| 固定入口 UI | 至少一处（列表行操作或会话顶栏）；触发状态层更新。 |
| 路由 | 不承担持久化；仅消费 `employee` 查询参数。 |

**插入位置（建议）**：`StudioMenuSection` **之下**、工作计划区**之上**，使「常用员工」高于「任务/历史」噪声；若产品评审反对，可调整但须两处侧栏一致。

## 7. 展示与降级策略

| 场景 | 策略 |
|------|------|
| id 在偏好中但不在当前 `getDigitalHumanList()` 结果中 | **不展示**该项，或展示占位并提示「已失效」（推荐前者以免误导点击）。 |
| 列表接口失败 | 偏好 id 可暂存；展示降级为仅 id 缩写或隐藏整块并 toast。 |
| 侧栏折叠 | 与现有工作室扩展区块一致：**折叠时不展开列表**（避免与钉应用区交互模型冲突）。 |
| 国际化 | 区块标题、空态、固定/取消 tooltip、`message` 文案需中英（及项目已支持语言）覆盖。 |

---

## 8. 错误、安全与配额

- **鉴权**：偏好接口仅在有效 Bearer token 下可用；与 dip-hub 其它受保护路由一致。
- **权限**：不假设「固定」即「管理」；仅表示快捷入口。若某员工对当前用户不可见，展示层过滤（§7）。
- **配额**：列表长度上限由服务端约束；客户端在追加前可软性提示将超出上限。

## 9. 测试要点（验收层）

| 层级 | 要点 |
|------|------|
| 契约 / 服务 | 有序去重、条数上限、非法 id 长度；PUT 全量替换语义。 |
| 前端状态 | pin 幂等；unpin 后列表一致；token 变更后重新拉取。 |
| E2E / 手工 | 点击固定项 URL 带正确 `employee`；新会话无错误 `sessionKey` 串联旧会话。 |
| 回归 | `HomeSider` 与 `AdminSider` 行为一致；折叠 / 展开无布局错位。 |
